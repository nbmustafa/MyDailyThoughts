## 🏗️ Platform Architecture

### What's been built

**5 core modules:**

**1. Platform Overview (Dashboard)**
Real-time workload table across all namespaces, HPA status, pipeline run history with stage visualization, per-namespace CPU/memory quota bars, and a live alert feed.

**2. Deploy Workload (The Core Abstraction)**
Tenants fill out a simple form — name, image, port, resources — and the platform auto-generates 3 production-hardened manifests behind the scenes:
- `Deployment` with `securityContext`, probes, topology spread constraints, IRSA ServiceAccount, and Prometheus annotations
- `HPA` (v2) with CPU + memory scaling targets
- `Ingress` with AWS ALB annotations, SSL redirect, and target-type IP
The YAML preview updates live so experienced engineers can still verify and export.

**3. Platform AI Advisor (Claude-powered)**
Tenants can ask natural language questions like *"what CPU limits for a Node.js API at 500 rps?"* — Claude gets the workload context injected automatically and responds with precise, EKS-specific recommendations.

**4. Observability**
Aggregated p99 latency, error rate, throughput, and restart counts. Live log stream across namespaces with severity coloring. Active alerts panel with PagerDuty-style severity tiers.

**5. Tenant Management**
Multi-tenant namespace isolation — quotas, network policy posture (Strict/Permissive), and team ownership — all in one view.

### Design Principles Applied
| Concern | Platform Handles It |
|---|---|
| `securityContext` | Auto-applied (runAsNonRoot, fsGroup) |
| Resource requests/limits | Set via sliders, requests = 50% of limits |
| Topology spread | Zone-aware scheduling enforced by default |
| Observability | Prometheus scrape annotations always injected |
| Multi-tenancy | IRSA ServiceAccount per namespace, NetworkPolicy |
| Rollout safety | RollingUpdate default, Canary/Blue-Green via Argo |
